## Тип: код (Flask + vanilla JS)

## Резюме
Проект хорошо структурирован и в целом корректен, но содержит несколько блокирующих проблем: отсутствие CSRF-защиты на формах, небезопасный fallback SECRET_KEY, уязвимость в `post_score` к манипуляции значением очков, и одна ошибка в порядке фильтрации в leaderboard_service.

---

## Проблемы

### КРИТИЧЕСКИЕ

- **[высокая]** `config.py`, строка 5 — Хардкоженный fallback `'dev-only-insecure-key'` попадёт в продакшн если переменная окружения не задана. Flask использует SECRET_KEY для подписи сессий. Если ключ известен или предсказуем — сессии можно подделать.
  Исправление: убрать fallback совсем и добавить явный `raise` если ключ не задан, либо добавить отдельный `ProductionConfig` без fallback.

- **[высокая]** `app/routes/auth.py` — Нет CSRF-защиты на формах `/register` и `/login`. Любой сторонний сайт может отправить POST-запрос от имени пользователя. Стандартное решение — flask-wtf с CSRFProtect, либо ручная генерация/проверка токена в сессии.

- **[высокая]** `app/routes/leaderboard.py`, строка 29 — `int(score_val)` без проверки диапазона. Клиент может передать отрицательное число или `999999999`. Серверная логика не проверяет, является ли переданный счёт достижимым значением для игры. Исправление: добавить проверку `if not isinstance(score_val, int) or score_val < 0 or score_val > MAX_POSSIBLE_SCORE`.

- **[высокая]** `app/routes/leaderboard.py`, строка 29 — `int(score_val)` упадёт с `ValueError` если `score_val` — строка, не приводимая к числу (например `"abc"`). Выше проверяется только `score_val is None`, но не тип. Нужно обернуть в `try/except ValueError` и вернуть 400.

### СРЕДНИЕ

- **[средняя]** `app/services/leaderboard_service.py`, строки 26-27 — Фильтр по `difficulty` применяется ПОСЛЕ `.order_by()`, но ДО `.limit()`. Это корректно с точки зрения SQLAlchemy (порядок вызовов не влияет на генерацию SQL), но вызов `.filter()` после `.order_by()` — плохая читаемость и потенциальный источник путаницы. Рекомендуется применять фильтр до `order_by`.

- **[средняя]** `app/__init__.py`, строка 31 — `User.query.get(int(user_id))` использует устаревший метод `.query.get()`, который депрекирован в SQLAlchemy 2.0. Следует заменить на `db.session.get(User, int(user_id))`.

- **[средняя]** `app/routes/auth.py` — Нет проверки минимальной длины пароля при регистрации. Пользователь может зарегистрироваться с паролем `"1"`. Исправление: добавить `if len(password) < 8` с flash-сообщением.

- **[средняя]** `app/routes/auth.py`, строка 11 — `username` обрезается через `.strip()`, но нет проверки максимальной длины. Если в БД колонка `String(64)`, можно передать строку длиннее 64 символов. SQLite молча обрежет, PostgreSQL бросит исключение. Добавить `if len(username) > 64`.

- **[средняя]** `app/routes/auth.py` — После логина нет редиректа на `next`-параметр. `login_manager` по умолчанию сохраняет исходный URL в `?next=`. Если не обрабатывать `request.args.get('next')`, пользователь после логина всегда попадает на `game.index`, теряя контекст.

- **[средняя]** `app/routes/leaderboard.py`, строка 10 — `difficulty` из query string передаётся напрямую в `get_top_scores()` и далее в `.filter(Score.difficulty == difficulty)`. SQL-инъекция исключена (ORM), но нет валидации значения на этапе роута. Можно передать `difficulty=foobar` и получить пустой список без ошибки — что само по себе не баг, но стоит возвращать 400 или игнорировать невалидные значения явно.

### НИЗКИЕ

- **[низкая]** `app/models/user.py`, строка 12 — `datetime.utcnow` депрекирован в Python 3.12+. Следует использовать `datetime.now(timezone.utc)` с импортом `timezone` из `datetime`.

- **[низкая]** `app/models/score.py`, строка 12 — та же проблема с `datetime.utcnow`.

- **[низкая]** `app/routes/auth.py`, строка 60-64 — `logout` доступен через GET-запрос. CSRF-уязвимость: достаточно `<img src="/auth/logout">` на стороннем сайте, чтобы разлогинить пользователя. Исправление: ограничить до POST или добавить CSRF-токен.

- **[низкая]** `app/static/js/themes.js`, строка 57 — `window.Themes = { init: init, switch: switchTheme }`. Ключ `switch` — зарезервированное слово в старых режимах JS, однако в ES5 strict mode как ключ объекта допустим. Практической проблемы нет в современных браузерах, но может вызвать путаницу.

- **[низкая]** `app/static/css/themes.css`, строка 47 — Селектор `:root.theme-cosmos body` не сработает корректно. `:root` — это `<html>`, и `body` не является дочерним элементом `:root` в контексте CSS-специфичности так, как ожидается. Технически селектор синтаксически валиден (`html.theme-cosmos > body`), но на практике в большинстве браузеров он сработает, т.к. `body` — дочерний элемент `html`. Низкий риск, но стоит протестировать.

- **[низкая]** `app/static/js/move_rating.js`, строки 10-11 — При `n >= 4` и `n === 3` оба возвращают `cls: 'rating-perfect'`, но label различается: `'★ PERFECT ★'` vs `'PERFECT'`. Это может быть намеренно, но выглядит как опечатка — возможно, для `n === 3` должен быть другой класс или другая метка.

## Интеграция frontend/backend

Прямого AUTH-объекта на фронтенде в переданных файлах нет (`move_rating.js` и `themes.js` — UI-утилиты без обращений к API). Если в проекте есть JS-код, который вызывает `/api/scores`, он должен:
1. Отправлять `Content-Type: application/json` (роут использует `get_json`).
2. Быть готов к ответу 401 при незалогиненном пользователе — `@login_required` вернёт редирект на `/auth/login`, а не JSON с ошибкой. Это сломает JS-клиент. Исправление: либо переопределить `login_manager.unauthorized_callback` для API-роутов, чтобы возвращал `jsonify({'status': 'unauthorized'}), 401`, либо добавить проверку `if not current_user.is_authenticated: return jsonify(...), 401` вручную.
